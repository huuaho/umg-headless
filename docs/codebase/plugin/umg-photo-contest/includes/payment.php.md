# docs/plugin/umg-photo-contest/includes/payment.php

**Purpose:** Payment-status REST endpoint and Stripe webhook receiver that marks users (individual flow) or applications (school flow) as paid.

## Responsibilities
Lets the frontend poll whether the logged-in user has paid the entry fee, and processes Stripe checkout settlement webhooks by matching the payer to a WP user (individual flow, from the static Payment Link) and marking them paid, **or** crediting every application in a school batch from one event (school flow, from a dynamically created Checkout Session — see [school.php](school.php.md)).

## REST routes (namespace `umg/v1`)
| Method | Path | Auth | Callback |
|--------|------|------|----------|
| GET | `/payment-status` | Bearer JWT | `umgpc_payment_status` |
| POST | `/stripe-webhook` | Stripe signature (`Stripe-Signature` header) | `umgpc_stripe_webhook` |

## Key exports
- `umgpc_payment_status(WP_REST_Request) -> response` — returns `{payment_status: 'paid'|'unpaid', payment_date: string|null}` from user meta.
- `umgpc_stripe_webhook(WP_REST_Request) -> response` — manually parses the `Stripe-Signature` header (`t`/`v1` parts), verifies HMAC-SHA256 of `"{timestamp}.{payload}"` against `UMGPC_STRIPE_WEBHOOK_SECRET` with `hash_equals`, rejects events older than 5 minutes (replay protection). Acts only on **settled** payments: `checkout.session.completed` with `payment_status === 'paid'` (immediate methods — cards, wallets) or `checkout.session.async_payment_succeeded` (asynchronous methods like Alipay, which complete first as "processing" and settle later). `async_payment_failed` and completed-but-still-processing sessions are acknowledged and skipped. Then branches on `metadata.purpose`: **`school_bulk_entry`** → delegates to `umgpc_mark_school_batch_paid()` (below) and returns immediately, before reaching any individual-flow logic; any **other non-`entry_fee`** purpose is skipped (acknowledged, not acted on); **empty or `entry_fee`** (the original path, unchanged) matches the WP user by `client_reference_id` (the WP user id, set by the frontend on the payment-link URL) first, falling back to `customer_email` (or `customer_details.email`) for sessions created before that param existed. On no match, `error_log`s the event id/type/email/amount/session/client_ref and still acknowledges with `{received: true}` so Stripe stops retrying — the miss is reconciliation data, not a signal to retry. On match, sets user meta `umgpc_payment_status = 'paid'`, `umgpc_stripe_payment_id` (session ID), `umgpc_payment_date`.
- `umgpc_mark_school_batch_paid($session) -> response` — reads `session.metadata.application_ids` (a comma-joined list of `umg_submission` post IDs, produced server-side by [school.php](school.php.md)'s `POST /school/checkout`, never client input), splits and casts to int, and for each one independently re-validates it's a real school-owned **submitted** post before crediting (`get_post_type`, `umgpc_school_batch === '1'`, `umgpc_status === 'submitted'`) — never trusts the metadata blindly even though it's server-generated. Sets the same three payment fields as the individual flow but as **post meta** on each valid application (`umgpc_payment_status`, `umgpc_stripe_payment_id`, `umgpc_payment_date`), all stamped from the one settlement event. Also releases [school.php](school.php.md)'s per-account checkout lock — resolved **once per batch** (from the first credited post's `umgpc_user_id`), and only released if `umgpc_school_checkout_lock_session` still matches this event's own session id (see Notes: a redelivered webhook must never clear a *different*, still-in-flight checkout's lock). Invalid IDs are skipped and logged; they don't fail the rest of the batch.

## Dependencies
- Internal: [jwt.php](jwt.php.md) (auth guard), [config.php](config.php.md) (`UMGPC_STRIPE_WEBHOOK_SECRET`), [school.php](school.php.md) (validates post IDs against `umgpc_school_batch`/`umgpc_status` meta it also writes).
- External: Stripe webhooks (signature verification implemented by hand — no Stripe SDK), WordPress user meta / post meta.

## Used by
- Frontend `getPaymentStatus` in [apps/umg/lib/auth/api.ts](../../../apps/umg/lib/auth/api.ts.md) (payment gate before submission, individual flow only).
- Stripe's webhook delivery system calls `/wp-json/umg/v1/stripe-webhook` for **both** flows: the individual flow's static Payment Link (carries `client_reference_id=<wp_user_id>` from [SubmissionForm.tsx](../../../apps/umg/app/photo-submission/components/SubmissionForm.tsx.md), `metadata.purpose=entry_fee` set on the link in the Stripe dashboard) and the school flow's dynamically created Checkout Session (carries `metadata.purpose=school_bulk_entry` + `metadata.application_ids`, created server-side by [school.php](school.php.md)'s `POST /school/checkout`, consumed by [ApplicationsCart.tsx](../../../apps/umg/app/school-registration/components/ApplicationsCart.tsx.md)'s "Pay" button).

## Notes
- Individual-flow payment is tracked **per user** (user meta) — one payment covers the user's single entry. School-flow payment is tracked **per application** (post meta) — one checkout can cover N applications, credited from a single settlement event.
- If a customer pays before ever requesting a login code, no WP user exists yet and the individual-flow webhook is a no-op — the payment is not retroactively applied when the user later signs up.
- Without `UMGPC_STRIPE_WEBHOOK_SECRET` set, every webhook returns 400 (both flows share this same signature check).
- A tampered `client_reference_id` could in principle credit the wrong user (individual flow); this matches the pre-existing trust model (the payment link is unauthenticated and email was equally forgeable) and is accepted for now — server-side association is planned for the custom-backend migration.
- Async settlement + `client_reference_id`/purpose matching were both added in response to a live outage investigation (Alipay payments failing at the Stripe↔Alipay partner layer, unrelated to this file) — see `docs/future-work/alipay-incident-log.md` and `docs/future-work/payment-pipeline-audit.md` in the repo root (not mirrored here; planning docs, not source).
- **School-batch branch added 2026-07-03, live-verified with a real payment** (not just a synthetic test): a 2-student batch was paid for in one $100 checkout and both applications flipped to `paid` with the identical `payment_date` timestamp, proving one settlement event correctly credits multiple applications. The individual-flow branch's regression safety was confirmed via `git diff` rather than a second real payment — the diff is purely additive (the new branch + one new function), zero lines changed in the pre-existing individual-flow code.
- **A double-Checkout-Session bug found by code review (2026-07-03), fixed mostly in [school.php](school.php.md)** — `POST /school/checkout` had no guard against creating two Checkout Sessions for the same unpaid batch (risking a school being charged twice with no reconciliation). The fix is a short-TTL lock set/checked in `school.php`; this file's role is releasing that lock once a batch is actually credited.
- **A follow-up review of that fix (2026-07-03, same day) found this file's own release logic had a bug:** the first version deleted the lock unconditionally on every credited post, with no check on *which* checkout set it. Since Stripe can redeliver an already-processed event, that could wipe out a different, later, still-in-flight checkout's lock for the same account — reopening the exact double-charge race the lock exists to prevent. Fixed by correlating the release to a specific session id (`umgpc_school_checkout_lock_session`, stamped by `school.php` when the session is created) and only running the resolution/release once per batch instead of once per post.
- **A third review round (2026-07-03, same day) found round 2's atomicity fix in `school.php` hadn't actually made anything atomic** (it computed whether the lock was acquired but never checked that before proceeding) **and that the lock's TTL was shorter than the async settlement window this very file's webhook explicitly supports** — both fixed in `school.php` (a proper lock-acquisition wrapper, and an explicit Stripe session `expires_at` that the lock TTL is derived from so it always outlives the session). This file's own change from that round: a log line for the currently-unreachable case where a credited post's `umgpc_user_id` is missing, so a stuck lock would at least be visible in logs rather than silent.
- **Architecture note:** an earlier plan for the school flow used N separate Payment Link payments (one per student) to avoid touching this file at all — rejected mid-build because it didn't deliver "one payment for the batch," the actual point of the feature. See `claude-context/current-work/bulk-registration/school-bulk-registration-plan.md` §0 for the full correction.
