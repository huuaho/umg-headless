# docs/plugin/umg-photo-contest/includes/school.php

**Purpose:** School/bulk-registration REST endpoints — full CRUD for multiple independent applications under one WordPress account, plus a combined Stripe checkout for the whole batch.

## Responsibilities
Lets one authenticated account (a school) own and manage **many** `umg_submission` posts at once — application create/list/get/update/delete, photo upload/remove, submit, retitle — where the individual-applicant flow in [draft.php](draft.php.md) assumes exactly one. Each school-created post is marked with `umgpc_school_batch = '1'` so it never collides with `draft.php`'s single-draft lookup. Also creates **one Stripe Checkout Session covering every submitted-unpaid application** the caller owns, so a school pays once for the whole batch rather than once per student — this is the actual point of the feature (see Notes). Student-proof upload is intentionally not offered here (waived for school applications per product decision). All-additive: no function in `draft.php`, `submission.php`, or `payment.php` is modified except one exclusion filter in `umgpc_find_draft_id()` (see [draft.php](draft.php.md) Notes) and one new branch in `payment.php`'s webhook (see [payment.php](payment.php.md)).

## REST routes (namespace `umg/v1`, all Bearer JWT)
| Method | Path | Callback |
|--------|------|----------|
| GET | `/school/applications` | `umgpc_school_list_applications` |
| POST | `/school/applications` | `umgpc_school_create_application` |
| GET | `/school/application/(?P<id>\d+)` | `umgpc_school_get_application` |
| PUT | `/school/application/(?P<id>\d+)` | `umgpc_school_update_application` |
| DELETE | `/school/application/(?P<id>\d+)` | `umgpc_school_delete_application` |
| POST | `/school/application/(?P<id>\d+)/photo` | `umgpc_school_upload_photo` |
| DELETE | `/school/application/(?P<id>\d+)/photo/(?P<mediaId>\d+)` | `umgpc_school_remove_photo` |
| POST | `/school/application/(?P<id>\d+)/submit` | `umgpc_school_submit_application` |
| POST | `/school/application/(?P<id>\d+)/retitle` | `umgpc_school_retitle_application` |
| POST | `/school/checkout` | `umgpc_school_create_checkout` |

## Key exports
- Helpers: `umgpc_school_with_named_lock($name, $timeout, $callback) -> mixed|WP_Error` (runs `$callback` only while holding a MySQL named lock — `GET_LOCK`/`RELEASE_LOCK` — scoped to the caller-supplied `$name`; fails **closed** with `WP_Error('lock_busy', ...)` if the lock can't be acquired within `$timeout`s — the callback literally cannot run without the lock, see Notes for why this matters), `umgpc_school_find_applications($user_id, $post_status = 'publish') -> int[]` (all `umg_submission` posts with matching `umgpc_user_id` AND `umgpc_school_batch = '1'`, oldest first; `$post_status` accepts an explicit array, e.g. `array('publish', 'trash')` — WordPress's `'any'` silently excludes `'trash'`/`'auto-draft'`, so it is not a safe way to mean "every status"), `umgpc_school_find_unpaid_submitted_applications($user_id) -> int[]` (same, further filtered to `umgpc_status = submitted` AND `umgpc_payment_status != paid` — the set a checkout should cover), `umgpc_school_next_seq($user_id) -> int|WP_Error` (atomically claims one sequence number — for a brand-new application or to backfill a legacy one, same code path either way; the single source of truth for `umgpc_school_seq`, see Notes for why it doesn't try to guarantee chronological ordering; callers must check `is_wp_error()`), `umgpc_school_get_owned_application($post_id, $user_id) -> int|WP_Error` (ownership check; always returns 404, never 403, so existence never leaks to a non-owner), `umgpc_school_build_summary($post_id) -> array` (`{id, division, first_name, last_name, status, payment_status}`), `umgpc_school_compute_title($post_id, $user_id) -> string|null|WP_Error` (computes `"School Application #N {Name} - {email}"`, backfilling via `umgpc_school_next_seq()` under a **per-post** named lock — never its own number, and never a bare check-then-write — if a pre-existing post never had one; `null` if no name stored yet, `WP_Error('lock_busy', ...)` if contention prevented claiming a number just now — see Notes for why these two are kept distinct).
- `umgpc_school_list_applications` — summaries of every application the caller owns.
- `umgpc_school_create_application` — creates a blank `umg_submission` post, stamps `umgpc_user_id`, `umgpc_status = draft`, `umgpc_school_batch = '1'`, `umgpc_payment_status = unpaid`, and a per-account sequence number `umgpc_school_seq` (used in the placeholder title `"School Application #N - {email}"` until a name is entered), claimed via `umgpc_school_next_seq()` (see Helpers). Returns `{id}`.
- `umgpc_school_get_application` — full detail, same field shape as `draft.php`'s `umgpc_get_draft` minus `student_proof`, plus `payment_status`/`payment_date` (post meta, not the account-level meta `draft.php`/`payment.php` use for the individual flow).
- `umgpc_school_update_application` — upsert, same field list and photo-metadata matching as `umgpc_save_draft`; also retitles to `"School Application #N {Name} - {email}"` once a name is known (via `umgpc_school_compute_title`). 400 `already_submitted` once finalized.
- `umgpc_school_retitle_application` — recomputes and applies the title from current fields. Deliberately **bypasses the `already_submitted` edit lock** (cosmetic metadata only, never application content) so old titles can be fixed retroactively. Propagates a `WP_Error('lock_busy', ...)` from `umgpc_school_compute_title()` as a real 429 response, distinct from `{retitled: false}` (no name yet) — see Notes.
- `umgpc_school_delete_application` — hard-deletes the post and any attached photo attachments. 400 `already_submitted` if finalized (submitted applications are never deletable via this API).
- `umgpc_school_upload_photo` / `umgpc_school_remove_photo` — same validation as `draft.php`'s photo endpoints (JPEG only, 20MB max, 3 slots), scoped to a specific application id instead of find-or-create. Guards against `finfo_open()` returning `false` (fixed 2026-07-03 — see Notes) before passing it to `finfo_file()`.
- `umgpc_school_submit_application` — mirrors `umgpc_submit_entry` ([submission.php](submission.php.md)): no server-side field validation, just a status flip + `umgpc_submitted_at` timestamp, scoped by ownership.
- `umgpc_school_create_checkout` — resolves the caller's own submitted-unpaid applications (server-side, never client-supplied), 400 if none; **then atomically checks-and-sets a per-account checkout lock** via `umgpc_school_with_named_lock()` (user meta `umgpc_school_checkout_lock_at`, TTL `UMGPC_SCHOOL_CHECKOUT_LOCK_TTL`), 429ing with `checkout_in_progress` if one is already active, and 429ing with `lock_busy` if the mutex itself couldn't be acquired (never silently proceeding unprotected — see Notes). Calls Stripe's Checkout Session API (`POST https://api.stripe.com/v1/checkout/sessions` via `wp_remote_post`, `Authorization: Bearer {UMGPC_STRIPE_SECRET_KEY}`) with `quantity` = the computed count, an explicit `expires_at` (`UMGPC_SCHOOL_CHECKOUT_SESSION_TTL` + buffer, rather than trusting Stripe's 24h default — see Notes), `metadata.purpose = school_bulk_entry`, `metadata.application_ids` = a comma-joined list of the covered post IDs, and `success_url`/`cancel_url` back to `/school-registration` (validated against `umgpc_allowed_origins()`, never trusting the request's `Origin` header blindly). On success, stamps which session owns the lock (`umgpc_school_checkout_lock_session` user meta) so [payment.php](payment.php.md)'s webhook only releases a lock it actually created. Releases the lock immediately on any Stripe-call failure so a failed attempt doesn't block a legitimate retry; on success it's released by the webhook once the batch is actually paid (or simply expires after the TTL if the checkout is abandoned). Returns `{url, application_ids, quantity, total}`; the frontend redirects the browser to `url`. Product name/description/image and `billing_address_collection`/`phone_number_collection`/`automatic_tax` are matched to the existing individual-flow Payment Link (read directly via Stripe CLI, not guessed) for a consistent checkout experience; `metadata.purpose` is deliberately **not** aligned to the individual flow's `entry_fee` — that's the load-bearing distinction [payment.php](payment.php.md)'s webhook uses to route between the two crediting paths.

## Dependencies
- Internal: [jwt.php](jwt.php.md) (auth guard), [post-types.php](post-types.php.md) (CPT), [config.php](config.php.md) (`UMGPC_STRIPE_SECRET_KEY`, `umgpc_allowed_origins()`), reuses `umgpc_build_photos_array()` from [draft.php](draft.php.md) (function, not the draft-lookup logic).
- External: WordPress media pipeline (`wp_handle_upload`, `wp_insert_attachment`, `wp_generate_attachment_metadata`, `wp_delete_attachment`), `WP_Query`, PHP `finfo`, WordPress HTTP API (`wp_remote_post` — outbound call to Stripe's Checkout Session API).

## Used by
Frontend `apps/umg/app/school-registration/` (`ApplicationsCart.tsx` for CRUD + the "Pay total" button, `SchoolApplicationForm.tsx` for the create/edit form) via `apps/umg/lib/school/api.ts`. [payment.php](payment.php.md)'s webhook validates application IDs against the same `umgpc_school_batch`/`umgpc_status` meta this file writes.

## Notes
- Post meta written: `umgpc_user_id`, `umgpc_status`, `umgpc_school_batch`, `umgpc_school_seq`, `umgpc_payment_status`, `umgpc_stripe_payment_id`, `umgpc_payment_date` (the latter two set by [payment.php](payment.php.md)'s webhook, not this file), all the same text/consent/photo meta keys `draft.php` uses. User meta: `umgpc_school_next_seq` (monotonic per-account counter), `umgpc_school_checkout_lock_at` / `umgpc_school_checkout_lock_session` (checkout-in-progress lock timestamp + which Stripe session owns it).
- **Six rounds of code review, same day (2026-07-03), progressively hardening the checkout lock and sequence-numbering logic** — kept in detail below since this file is payment-adjacent and the history explains *why* the current code looks the way it does, not just what it does:
  - **Round 1** found three real bugs: no guard against two Checkout Sessions covering the same batch (double-charge risk); `umgpc_school_seq` computed as a live `count()` of existing applications, colliding after any deletion; unguarded `finfo_open()` failure crashing uploads with a 500 (also fixed in [draft.php](draft.php.md), which this file's upload validator was copied from).
  - **Round 2** found round 1's own fixes were check-then-set, not atomic — two near-simultaneous requests could both pass the guard before either wrote it, for both the checkout lock and the sequence counter. Also found round 1's webhook fix released the checkout lock unconditionally on any credited batch, letting a redelivered Stripe webhook wipe out a *different*, still-in-flight checkout's lock. Backfilled pre-existing accounts' counters too (round-1 gap).
  - **Round 3** found round 2 had computed whether `GET_LOCK` succeeded but never actually gated on it — fail-open, not fail-closed, under contention. Fixed by extracting `umgpc_school_with_named_lock()` (see Helpers): the critical section is a callback that structurally cannot run unless the lock was acquired. Also found the checkout lock's TTL (300s) was shorter than the async settlement window `payment.php` explicitly supports (Alipay-style), and that the new atomic counter disagreed with `umgpc_school_compute_title()`'s own independent legacy backfill. Fixed the TTL by deriving it from an explicit Stripe `expires_at` rather than trusting Stripe's 24h default, and unified the two backfill paths into one.
  - **Round 4** confirmed the lock construction itself now held up, but found the retitle endpoint couldn't distinguish "no name yet" from "lock busy, try again" (both collapsed to the same response), and that trying to force perfectly chronological sequence numbering by batch-backfilling every un-numbered sibling at once broke on visit-order dependence. Also investigated and ruled out a theoretical Stripe `expires_at`/async-payment-method incompatibility (checked directly against Stripe's docs — none exists), and fixed an arithmetic drift between the lock TTL and the actual Stripe expiry.
  - **Round 5** found the round-4 batch-backfill redesign had introduced four more bugs, all stemming from trying to force strict chronological order across a mix of legacy and newly-created applications. **Rather than patch a sixth time, `umgpc_school_next_seq()` was simplified**: dropped batch-backfill entirely, back to one atomic claim per call (same path for a new application or a legacy backfill) — `umgpc_school_seq` is a purely cosmetic wp-admin label, never used for payment matching or any other load-bearing logic, so only **uniqueness** and **atomicity** are treated as real requirements; perfectly chronological ordering across mixed legacy/new applications was deliberately given up as a cosmetic, non-bug limitation.
  - **Round 6** confirmed the simplified counter's atomicity, but found two narrower real issues in that simplification: the "scan every post status" backfill used `post_status => 'any'`, which WordPress silently excludes `'trash'` from — so a trashed-then-restored numbered application could have its number reused. Fixed by reusing `umgpc_school_find_applications()` with an explicit status list (now accepts a `$post_status` param) instead of a second hand-written query. Also found `umgpc_school_compute_title()`'s per-post check-then-write of *its own* `umgpc_school_seq` wasn't itself lock-protected (distinct from the per-account counter, which was) — two concurrent calls for the *same* post could each claim a distinct number and race on which write landed last. Fixed with a per-post named lock (`umgpc_school_seq_post_{id}`) around that specific check-claim-write, re-checking inside the lock so a request that had to wait just returns whatever number a concurrent request already assigned, rather than wasting a second one.
  - **Round 7** confirmed the per-post lock's nesting (per-post lock wrapping a call into `umgpc_school_next_seq()`'s separate per-account lock) is safe on any MySQL/MariaDB version WordPress itself already requires (documented in `umgpc_school_with_named_lock()`'s own docblock — only unsafe on pre-5.7.5 MySQL, not a real concern on any supported host). Found one real remaining gap: the round-6 fix only passed `array('publish', 'trash')` to the scan, not truly "every status" as its own comment claimed — a post manually moved to `'draft'`/`'pending'` via wp-admin Quick Edit (a normal action on this CPT) would still be missed. Fixed by using `get_post_stati()` (every status WordPress has registered) instead of a hardcoded list, so the code now actually matches what it claims to guarantee.
- **Live-tested end to end, including a real payment (2026-07-03).** Backend CRUD: create/list/get/update/delete round-trip, ownership isolation (a second real account gets clean 404s on both GET and DELETE of another's application), the 3-photo cap, post-submission lock on edit/delete/photo-upload, and the core multi-student scenario (one account with several independently tracked applications, mixed draft/submitted, including pausing and resuming one mid-way). Payment: built a 2-student batch, paid $100 in one real Stripe checkout, both applications flipped to `paid` with the **identical `payment_date` timestamp** — the concrete proof that one checkout credits multiple applications from a single settlement event, which is the entire reason this feature exists. Refunded afterward (Stripe's ~3.5% processing fee is not returned on refund).
- **Architecture note — why a Checkout Session, not the static Payment Link.** An earlier plan for the payment step used N separate clicks through the existing individual-flow Payment Link (one $50 payment per student) specifically to avoid adding new checkout-creation code under a same-day deadline. That was rejected mid-build: it doesn't deliver "one payment for the batch," which is the actual point of the feature (the client's original question was how schools avoid needing a separate payment per student). The Checkout Session approach lets Stripe itself compute and enforce the total (`price × quantity`), so there's no reconciliation risk the way there would be with a customer-editable-quantity Payment Link. Full writeup: `claude-context/current-work/bulk-registration/school-bulk-registration-plan.md` §0.
- **A real bug was caught and fixed during live payment testing:** `wp_remote_post()`'s array body is serialized via PHP's `http_build_query()`, which turns a PHP `true` into the string `"1"` — Stripe's form-encoded parser rejects `"1"` as an invalid boolean and requires the literal word. Broke `automatic_tax.enabled` and `phone_number_collection.enabled`. Fixed by using the literal strings `'true'`/`'false'` for Stripe boolean fields, never PHP booleans, when building the `wp_remote_post` body. Found via Stripe's own live request-log tail, which confirmed the request *was* reaching Stripe (a 400 response), ruling out an earlier hosting/network-block theory.
- A second bug (unrelated to this file, but found via a `division` value this file lets be freely set) is documented in the frontend's `SchoolApplicationForm.tsx` doc: an invalid `division` string crashed the read view before a validation fallback was added.
- Full design rationale, all confirmed product decisions, and the day-one-stopgap-vs-real-design history: `claude-context/current-work/bulk-registration/school-bulk-registration-plan.md`. Commit-by-commit build log with every test performed: `claude-context/current-work/bulk-registration/implementation-checklist.md`.
